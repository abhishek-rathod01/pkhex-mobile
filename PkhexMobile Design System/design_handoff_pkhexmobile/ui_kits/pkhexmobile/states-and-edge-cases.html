PkhexMobile — component states & edge cases

Reference sheet for handoff: every interactive control's default / hover / pressed / disabled treatment, plus how the party list and detail screens degrade for missing or generation-specific data. Static swatches are colour-accurate to the live components; each grid ends with a live, interactive instance to confirm behaviour by hand.

Interactive states — Button
4 variants × 4 states. Hover tints the fill/surface; pressed adds scale(0.97); disabled drops to 50% opacity and blocks pointer events.
Live — hover & click to confirm
Interactive states — IconButton
3 variants × 4 states. Pressed scale is tighter (0.92) than Button since the hit target is smaller.
Live — hover & click to confirm
Interactive states — Switch
Off / On × enabled / disabled. Disabled keeps the last value's position but flattens to grey at 60% opacity — the state is still legible, just inert.
Live — click to toggle
Save button — disabled-when-clean / enabled-when-dirty
The primary Save action (bottom action bar, section edit sheets) is disabled the instant a screen loads with no unsaved changes, and enables the moment any field is edited. This prevents accidental no-op writes to the save file. Toggle the switch below to simulate an edit.
Interactive states — party row & collapsible section
Rows and section headers use a subtle sunken-tint + scale press instead of a hover fill, so the list reads calmly while still confirming touch.
Held-item display
Two contexts render the held item differently: the party row shows a compact inline Bag icon + first word of the item name (space-constrained); the detail screen's Main card shows the full name in a Chip. Both share the same muted "no item" treatment when empty.
